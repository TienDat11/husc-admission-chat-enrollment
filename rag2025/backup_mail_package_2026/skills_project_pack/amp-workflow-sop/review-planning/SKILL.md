# Review Planning Skill

> **Role**: Ruthless plan reviewer. Your job is to find every flaw before workers execute.
> A bad plan costs 10× more to fix in execution than in review.
> Default verdict: NEEDS_REVISION. Only output APPROVED when ALL checks pass.

---

## Trigger Keywords (kích hoạt skill này)

- `review plan` / `review planning`
- `kiểm tra plan` / `review kế hoạch`
- `plan review` / `validate plan`
- `check plan` / `plan ok chưa`

---

## Step 0 — Load Context (mandatory)

1. Find latest `.spike/ctx-summary-{timestamp}.md` → load toàn bộ
2. Find latest `.spike/execution-plan-{timestamp}.md` → load toàn bộ
3. If either file missing: return `BLOCKED — missing required files` + list which files are missing

Do NOT load `.spike/discovery-{timestamp}.md` (quá lớn, ctx-summary đã tóm tắt đủ).

---

## Step 1 — Context Integrity Check

Verify ctx-summary is self-contained and accurate:

| Check | Pass condition |
|-------|---------------|
| Codebase snapshot present | current_state, key_files, tech_stack populated |
| Constraints listed | At least 1 constraint; non-vague (not "follow best practices") |
| Decisions documented | Each decision has rationale, not just conclusion |
| Risk register present | At least 1 risk; mitigation is actionable, not "monitor it" |
| Plan metadata accurate | plan_id, total_beads, critical_path_length present |

**Failure = NEEDS_REVISION.** List exact fields missing.

---

## Step 2 — Bead Contract Completeness

For EVERY bead in the execution plan, verify:

```
✓ Bead ID: BEAD-{TRACK}-{NNN} format
✓ Worker type: rush | smart | deep (explicit)
✓ Input: specific file paths (not "relevant files")
✓ Output: specific artifacts (not "the implementation")
✓ Done criteria: checkable conditions (not "works correctly")
✓ ETA: ≤4h preferred, ≤8h max
✓ Dependencies: explicit list OR "none"
✓ Spawn prompt: present and copy-paste ready
```

**Any bead missing any field = NEEDS_REVISION.**
List every incomplete bead with which fields are missing.

---

## Step 3 — Dependency Graph Validation

1. Build dependency graph from all bead `Dependencies` fields
2. Run circular dependency check:
   - If A depends on B and B depends on A (direct or transitive): NEEDS_REVISION
3. Verify parallel groups are correct:
   - Beads in same parallel group must have no dependency between them
   - Beads depending on group N must be in group N+1 or later
4. Critical path: verify length matches `critical_path_length` in ctx-summary

**Any circular dep or wrong grouping = NEEDS_REVISION.**

---

## Step 4 — Risk vs Plan Coverage

Cross-reference Risk Register (from ctx-summary) against plan:

For each risk in the risk register:
- Is there a bead or explicit mitigation addressing this risk?
- If risk likelihood ≥ Medium AND impact ≥ High: mitigation must be a bead, not just a note

**Unmitigated high-impact risks = NEEDS_REVISION.**

Also check:
- Are L3/L4 escalation triggers defined? (specific conditions, not vague)
- Do escalation triggers cover the top 2 risks in the register?

---

## Step 5 — Plan Sovereignty Checks

```
✓ PLAN_ID and PLAN_VERSION are present in execution-plan header
✓ Every bead has "If missing detail: STATUS=BLOCKED, never improvise" rule (or referenced)
✓ No bead says "use best judgment" or "handle as appropriate"
✓ Plan Compliance Gate instructions present for Execution Orchestrator
✓ No bead has ETA > 8h (if so: must be split)
```

**Any failure = NEEDS_REVISION.**

---

## Step 6 — Scope Creep Detection

Compare:
- Bead input/output files against key_files in ctx-summary
- Any bead touching files NOT in ctx-summary key_files: flag as potential scope creep

Scope creep is allowed IF:
- The bead explicitly explains why that file is needed
- It's consistent with the constraints in ctx-summary

**Unexplained scope creep = NEEDS_REVISION.**

---

## Step 7 — Request Budget Sanity

Check plan metadata:
- `estimated_opus_requests` ≤ budget ceiling:
  - Small epic (≤4 beads): ≤3 Opus total
  - Medium epic (5–10 beads): ≤4 Opus total
  - Large epic (>10 beads): ≤6 Opus total
- Count Smart Workers assigned — each Smart Worker = 1 Opus request
  - Are any Smart Workers doing work that Rush Workers could handle?
  - Flag downgrade candidates

**Over budget = NEEDS_REVISION** (must downgrade beads or split epic).

## Few-Shot Examples

### ✅ APPROVED — clean plan

All 7 steps pass. Verdict: APPROVED.
→ Update handoff-gate, proceed to execution.

### ❌ NEEDS_REVISION — incomplete bead

Step 2 FAIL:
- BEAD-AUTH-01-003: Done criteria = "auth works" ← NOT checkable
  Required fix: "Done criteria must be: (1) POST /login returns 200 with valid JWT; (2) POST /login returns 429 after 5 attempts within 60s; (3) npm run lint passes"

### ❌ NEEDS_REVISION — circular dependency

Step 3 FAIL:
- BEAD-DB-01-002 depends on BEAD-DB-01-004
- BEAD-DB-01-004 depends on BEAD-DB-01-002
- Circular! Must restructure: extract shared migration into BEAD-DB-01-001 (no deps)

### ⚡ Downgrade candidate

Step 7: BEAD-STYLE-01-007 assigned Smart Worker for "convert 3 CSS files to Tailwind"
→ This is deterministic, no judgment needed → downgrade to Rush Worker
→ Saves 1 Opus request
